Walacor Gateway

AI Security & Governance Proxy — Architecture & Workflow

What It Is

What we are

  • Security & compliance proxy for LLM traffic
  • Like a WAF (Web Application Firewall) for AI
  • Transparent — apps don't change their code
  • Tamper-evident audit trail (ID-pointer chain + Walacor DH)
  • Content safety scanner (PII, toxicity, Llama Guard)
  • Tool execution engine for local models
  • Provider-agnostic (Ollama, OpenAI, Anthropic, etc.)

What we are NOT

  • Not a model wrapper — we don't host or modify models
  • Not an LLM framework (not LangChain/LlamaIndex)
  • Not a chatbot or AI application
  • Not a load balancer
  • Not a model training platform

Request Pipeline (8 Steps)

1
Authenticate Caller
Validate API key or JWT token. Resolve caller identity (user, team, roles, email) for audit trail.
Valid key/JWT → proceed Invalid → 401 Unauthorized
auth/api_key.py • auth/jwt_auth.py • auth/identity.py
2
Model Attestation (G1)
Is this model registered and approved? Check attestation cache. Auto-attest on first use if no control plane. Block if model is revoked.
Active → proceed Revoked → 403 Forbidden Unknown + no CP → auto-attest
pipeline/model_resolver.py • control/store.py
3
Pre-Inference Policy (G3)
Evaluate policy rules: who can use which models, from which teams, under what conditions.
Policy pass → proceed Policy deny → 403 Forbidden
pipeline/policy_evaluator.py
4
Token Budget Check
Has this tenant/user exceeded their daily or monthly token limit? Pre-reserve estimated tokens before the call.
Budget OK → proceed Exhausted → 429 Too Many Requests
pipeline/budget_tracker.py
5
Forward to LLM Provider
Send request to Ollama/OpenAI/Anthropic. For active tool strategy: inject tool definitions, override to non-streaming. Auto-detect model capabilities (tool support).
pipeline/forwarder.py • adapters/ollama.py • adapters/openai.py
6
Tool Execution Loop
If model requests tool calls (web search, MCP tools): gateway executes them, scans output for safety, hashes input/output, feeds results back to model. Repeats until model gives a final answer.
Tool results safe → feed to model Prompt injection detected → block tool output
pipeline/orchestrator.py • tools/web_search.py • mcp/registry.py
7
Post-Inference Content Analysis (G4)
Scan the model's response through 3 analyzers before returning to the client.
PII (credit card, SSN, API key) → BLOCK PII (email, phone, IP) → WARN Llama Guard S4 (child safety) → BLOCK Llama Guard S1/S9/S11 → WARN Toxicity → BLOCK
content/pii_detector.py • content/toxicity_detector.py • content/llama_guard.py
8
Audit Record + Session Chain (G5)
Build execution record with all metadata. Link to previous record via ID-pointer (record_id + previous_record_id). Sign canonical ID string with Ed25519. Dual-write to Walacor ledger + local WAL — Walacor returns a tamper-evident DH digest on ingest. Emit OTel span.
Record written → tamper-proof audit trail
pipeline/hasher.py • pipeline/session_chain.py • wal/writer.py

How the Gateway Manages MCP & Tool Calls

The gateway does not invent tool calls. The model decides when to call a tool. The gateway's job is to make tools available, execute them safely, and audit everything.

Phase 1: Startup — Tool Discovery

When the gateway starts, it connects to all configured tool sources and builds a unified catalog.

Source 1: MCP Servers (external tools)
Configured via WALACOR_MCP_SERVERS_JSON
[{ "name": "my-db",
   "command": "npx",
   "args": ["mcp-server-postgres",
            "postgres://..."] }]
Gateway starts subprocess → connects via MCP protocol → calls list_tools() → discovers available tools
mcp/client.py • mcp/registry.py
Source 2: Built-in Tools (e.g. web search)
Enabled via WALACOR_WEB_SEARCH_ENABLED=true
// WebSearchTool duck-types MCPClient
get_tools() → [{ name: "web_search" }]
call_tool("web_search", {query: ...})
  → calls DuckDuckGo / Brave / SerpAPI
No subprocess needed. Same interface as MCP — the registry doesn't know the difference.
tools/web_search.py
Unified Tool Registry
Both sources register into ToolRegistry — a single catalog with one execute_tool(name, args) method.
The orchestrator doesn't care if a tool is MCP or built-in.
mcp/registry.py

Phase 2: Request Time — The Tool Loop

When a request comes in and the model is local (Ollama), the gateway manages the full tool lifecycle:

A
Inject Tool Definitions
Gateway reads all tools from ToolRegistry, converts them to OpenAI function-calling format, and adds them to the request body before sending to Ollama. The model now knows what tools are available.
// Gateway adds this to the request body:
"tools": [{ "type": "function", "function": {
  "name": "web_search",
  "description": "Search the web for current information...",
  "parameters": { "query": { "type": "string" } }
}}]
orchestrator.py:282-301 • _inject_tools_into_call()
B
Model Responds with Tool Call (not an answer)
The model decides on its own whether to use a tool. If it needs information, it responds with finish_reason: "tool_calls" instead of a regular answer.
// Ollama says: "I want to call this tool"
{ "finish_reason": "tool_calls",
  "tool_calls": [{ "function": {
    "name": "web_search",
    "arguments": "{\"query\": \"Python programming\"}"
  }}]
}
The model CANNOT execute this itself. It's just text output saying "I want to search."
C
Gateway Intercepts and Executes the Tool
The gateway sees finish_reason: tool_calls, looks up the tool in ToolRegistry, validates the arguments against the tool's schema, and executes it.
// Gateway does this:
1. Validate args against schema (required fields present?)
2. registry.execute_tool("web_search", {query: "Python"})
3. Registry dispatches to WebSearchTool (or MCPClient)
4. WebSearchTool calls DuckDuckGo API → gets results
5. Returns ToolResult { content, sources, duration_ms }
orchestrator.py:422-479 • _execute_one_tool()
D
Scan Tool Output for Safety
Before feeding tool results back to the model, the gateway runs all content analyzers on the tool output. This catches indirect prompt injection — a malicious website could embed instructions in its content to manipulate the model.
Clean output → feed to model PII/toxicity/injection → replace with "[blocked by content policy]"
orchestrator.py:456-471
E
Record Tool Event
Gateway records the tool's input and output verbatim (Walacor backend hashes on ingest), captures source URLs, duration, and content analysis results. This becomes a first-class audit event.
{ "tool_name": "web_search",
  "input_data": { "query": "Python programming" },
  "output_data": "...verbatim tool output...",
  "sources": [{ "title": "Python.org", "url": "..." }],
  "content_analysis": [{ "verdict": "pass" }],
  "duration_ms": 342 }
orchestrator.py:336-372 • _build_tool_event_record()
F
Feed Results Back to Model → Get Final Answer
Gateway builds a new request with the tool results injected as a "tool" role message and sends it back to the model. The model now generates its final answer using the tool data. If the model requests more tools, the loop repeats (steps B-F).
// Gateway sends this back to Ollama:
{ "role": "tool",
  "tool_call_id": "call_abc123",
  "content": "{ results: [{ title: \"Python.org\", ... }] }" }
orchestrator.py:482-532 • _run_active_tool_loop()
Key Insight: The Model Never Touches the Outside World
The LLM only sends text that says "I want to call web_search with query=X". It has no network access, no API keys, no database credentials. The gateway is the only thing that actually executes tools, and it does so through a controlled pipeline:
Step 1
Validate Arguments
Check required fields against tool schema
Step 2
Execute with Timeout
30s enforced timeout per tool call
Step 3
Scan Output
PII + toxicity + Llama Guard
Step 4
Record Everything
Verbatim input + output stored for audit

Phase 3: Smart Model Detection

Not all models support tool calling. The gateway automatically detects this and adapts.

qwen3:1.7b — supports tools
1st request: Gateway injects tools → model accepts → gateway caches supports_tools=True
2nd+ request: Gateway knows tools work → injects tools → full tool loop available
gemma3:1b — does NOT support tools
1st request: Gateway injects tools → model returns 400 "does not support tools" → gateway strips tools, retries, caches supports_tools=False
2nd+ request: Gateway skips tool injection entirely → no wasted round-trip
orchestrator.py:55-68 • _model_capabilities registry
